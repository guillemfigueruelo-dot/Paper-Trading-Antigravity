# Handoff Report

## 1. Observation
- In `bot/trading/engine.py` (line 22), `base_allocation` is calculated as 10% of `portfolio["USD"]` *before* the loop over trade decisions begins.
- The `for symbol, decision in decisions.items():` loop processes decisions (line 27) in arbitrary order, meaning `SELL` and `BUY` actions are evaluated sequentially as they appear in the dictionary.
- If a `SELL` increases the local `usd_balance`, `base_allocation` remains unchanged, making the freed capital unavailable for `BUY` decisions in the same cycle (the "Zero-Start Bug & Reinvestment Failure").
- Existing tests (`bot/test_engine.py` and `bot/test_trade_logic.py`) predominantly test `BUY` actions in isolation and do not cover mixed `BUY`/`SELL` scenarios to catch this state dependency.

## 2. Logic Chain
- To fix the reinvestment failure, any `SELL` decisions must be executed *first* so that their proceeds can replenish the `usd_balance`.
- By splitting the trade processing into two passes—one for `SELL` decisions and one for `BUY` decisions—we can guarantee that all proceeds are collected before we decide how much to allocate for buying.
- Recalculating `base_allocation = float(portfolio["USD"]) * 0.10` *after* the `SELL` pass and *before* the `BUY` pass ensures that subsequent `BUY` trades will use 10% of the *new*, larger USD balance.
- `BUY`s are then processed sequentially, strictly using `min(base_allocation, usd_balance)` and updating local state/DB, which correctly caps allocation and prevents race conditions since all updates remain atomic/sequential.
- To prevent regression, the test suite must be updated to include a scenario demonstrating this two-pass behavior: starting with 0 USD, executing a `SELL`, and verifying that a subsequent `BUY` successfully uses the newly generated capital.

## 3. Caveats
- **Sequential Starvation:** The recommended strategy sets `base_allocation` to 10% of the updated USD balance. If there are >10 `BUY` signals, they will still fully deplete the USD balance, starving the 11th trade (it will get `min(base_allocation, 0) = 0`). This is a mathematically unavoidable consequence of a fixed 10% allocation rate unless we dynamically scale allocation by the number of `BUY` signals (e.g., `usd_balance / num_buys`). Since the recommended strategy explicitly specifies using the 10% logic, this limitation remains.
- **Concurrency:** The sequential nature of the script inherently prevents internal race conditions during execution, fulfilling the strict DB update atomicity required. However, it assumes this bot is the only writer to the DB portfolio at the time of execution.

## 4. Conclusion
Implement the two-pass processing strategy in `bot/trading/engine.py`:
1. Iterate over `decisions` and process all `SELL` actions, updating `portfolio["USD"]` and making DB calls.
2. Re-calculate `base_allocation = float(portfolio["USD"]) * 0.10`.
3. Iterate over `decisions` again to process all `BUY` actions sequentially (updating local balance and DB).
4. (Optional) Iterate over `decisions` to print `HOLD` and invalid actions for logging.
5. Update `tests/test_engine.py` and `tests/test_trade_logic.py` to include a mixed `BUY`/`SELL` test case that asserts a `BUY` can utilize capital generated by a `SELL` in the same cycle.

## 5. Verification Method
1. Modify `bot/trading/engine.py` according to the strategy.
2. Add a new test case `test_sell_then_buy_reinvestment` in `bot/test_trade_logic.py`. Mock the portfolio to start with `USD: 0.0` and `AAPL: 10.0`. Set a `SELL` decision for `AAPL` and a `BUY` for `GOOG`. 
3. Run `pytest bot/test_engine.py bot/test_trade_logic.py`.
4. The test passes if the `SELL` goes through, increases `USD` balance, and the `BUY` successfully executes with a `base_allocation` based on the new `USD` balance.
